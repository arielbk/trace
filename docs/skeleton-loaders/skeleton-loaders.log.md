# Skeleton loaders — implementation log

## `list-skeleton` — 2026-07-03 14:06:53

**Status:** done
**Summary:** Added `useSkeletonReveal(ready)` hook (`apps/web/src/components/useSkeletonReveal.ts`) driving the mount/unmount timing of a cross-fade skeleton, the `t-skel` CSS block + `--skel-*` custom properties in `index.css`, and a `TaskListSkeleton` (6 pulsing row placeholders) wired into `TasksPage` in place of the old `Loading...` string.
**Deviations:** The transitions-dev skill's literal `.t-skel-skeleton`/`.t-skel-content { position: absolute; inset: 0; }` assumes a fixed-size card. A task list's height varies with row count, so both layers would collapse to zero height while only the (variable) skeleton is mounted. Implemented instead: both layers sit in normal flow pre-reveal (skeleton visible, opacity 0 content underneath — content isn't even mounted while pending); once `.t-skel.is-revealed`, only the *skeleton* detaches to `position: absolute; inset: 0` and fades out, while the real content (now mounted) stays in flow and drives the container's height. Net visual effect (pulse → cross-fade) is identical; only the mechanism for who's "absolute" differs. `detail-skeleton` should reuse this same variant (its timeline height varies with item count too), not the literal skill snippet.
**Handoff:** `useSkeletonReveal(ready: boolean): { showSkeleton, revealed }` — `showSkeleton` true while the skeleton should be mounted (loading, or still fading out post-reveal), `revealed` is a direct passthrough of `ready` (flips instantly, no lag) used to toggle `.is-revealed` and to gate rendering of real content/subtitle/filter bar. Reveal window is a hardcoded `REVEAL_MS = 400` constant in the hook file — **must stay in sync with `--skel-reveal-dur` in `index.css`** (same pattern as `TaskRow`'s `ARCHIVE_EXIT_MS` comment). CSS classes to reuse for `detail-skeleton`: `.t-skel` / `.t-skel-skeleton` / `.t-skel-content` / `.t-skel-bar` (generic placeholder bar, background `var(--color-chip-bg)`) / `.is-pulsing` / `.is-revealed`, plus the `--skel-pulse-*` / `--skel-reveal-*` root tokens.
Also fixed an unrelated bug this slice's refactor exposed: `TasksPage` previously returned early (bare `<p>Loading...</p>`, no `<AppHeader>`) while pending, so `AppHeader`'s nested `ThemeToggle` never mounted during `App.test.tsx`'s synchronous test (which doesn't await the query and has no `matchMedia` stub). Now that the header renders unconditionally (even mid-load), `ThemeToggle`'s mount effect calls `window.matchMedia`, which crashed only in `App.test.tsx`. Fixed by adding the same `matchMedia` stub `TasksPage.test.tsx` already uses. Full suite (235 tests), typecheck, and lint are all green.

## `heading-fix` — 2026-07-03 14:12:53

**Status:** done
**Summary:** Fixed `TaskPage`'s header row (`TaskTimelineView` in `apps/web/src/pages/TaskPage.tsx`) so a long task title no longer forces the "Last active" block onto its own line. The row container had `flex flex-wrap items-start justify-between gap-5`; with `flex-wrap`, the flex-wrap algorithm uses each item's unshrunk hypothetical width to decide line breaks, so a long `h1` (which has no width constraint) forced the whole second flex item — the shrink-0 "Last active" block — down to a new line once the two items' natural widths exceeded the container. Removed `flex-wrap` from the row container and added `min-w-0` to the `h1` so it shrinks to the remaining space and wraps its own text across multiple lines instead, keeping "Last active" pinned top-right on the same row.
**Deviations:** none.
**Handoff:** Added one new automated test, `TaskTimelineView keeps Last active pinned to the heading row for a long title` in `TaskPage.test.tsx`, asserting the header row div doesn't carry `flex-wrap` and that the "Last active" text stays inside that same row element — this is a structural/class-based regression guard since jsdom doesn't do real layout, so it can't verify visual wrapping directly. Manual eyeballing of the cross-blur/wrap on a real long-titled task is still recommended but wasn't performed in this sandboxed iteration (no dev server / browser available). Full suite (236 tests), typecheck, and lint are all green. This slice was independent of `list-skeleton`/`detail-skeleton` — no shared code touched.
